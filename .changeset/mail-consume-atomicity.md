`totem mail reply` now marks the source dispatch processed in the same command as writing the reply, and a new `totem mail mark <source>` marks a consumed dispatch without replying (ADR-106 § A1.4; mmnto-ai/totem#2396). Previously the copy-into-`processed/` step lived on no command surface — the consuming seat had to remember to hand-copy the source into its `processed/` cursor, and a dropped copy re-surfaced the dispatch as phantom-unread on every subsequent poll (§ A1.3 names this the entire false-unread class). A mark is a same-basename copy of the source under the consuming seat's `processed/` (directed) or `processed/_broadcast/` (broadcast) — the exact stores the reader drains and `ecl-gc` compacts. Seat resolution is the single-writer `ecl-gc` discipline (explicit `--agent-id` > unambiguous resolved self > error): a seat can only mark into its OWN cursor, and a directed dispatch whose recipient is not the resolving seat is refused (its any-seat subtraction would otherwise suppress the real recipient's mail). Reply marking is on by default; `--no-mark` opts out for stage-only reply workflows. The reply and the mark are two sequential writes, not one filesystem-atomic operation: the reply lands first, and if the mark then fails the error names the written reply and routes recovery to `totem mail mark <source>` so the reply is never re-sent as a duplicate. Marking is idempotent (a second mark is a no-op) and otherwise fail-loud (an unreadable/unparseable source, an ambiguous seat, or a write that did not land is a hard error, never a silent drop).

Marks are stored under a colon-stripped, NTFS-safe basename (mmnto-ai/totem#2431): a colon in a filename is an NTFS Alternate Data Stream separator, so a colon-bearing legacy source name written verbatim would "succeed" into a 0-byte base file with the real bytes in an invisible stream `fs.readdirSync` never lists — the mark would exist yet never subtract, and `mail mark` would still report success. A shared `sanitizeEclBasename` normalizes the writer's output and both sides of the reader's comparisons — the unread subtraction, the broadcast/self-token filename classification, and the `ecl-gc` A2.1 retention + A2.4 resurfacing comparisons — so a sanitized mark subtracts its colon-bearing inbound name. A pre-existing corrupted (0-byte ADS) mark is superseded when a re-mark writes the sanitized copy; polling itself only normalizes comparisons and rewrites nothing. The send actuator's filename emitter was already colon-free (the compact `yyyyMMddTHHmmZ` stamp).
