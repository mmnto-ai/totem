## Matcher policy (binding, all mechanisms)

- **Presence invariant, zero semantics.** Any GitHub close-keyword
  (`close/closes/closed`, `fix/fixes/fixed`, `resolve/resolves/resolved`)
  immediately adjacent to an issue reference is an anomaly — genuine, negated,
  quoted, or emphasized alike. **No negation parser.**
- **Qualified refs** (`owner/repo#N`) still auto-close → match them too.
- **Never scan issue/PR comment bodies** — comments never auto-close.
- **ONE shared evaluator** in `packages/core/src/autoclose/`; D1, D2, and C's
  template all consume it. C's rendered `.cjs`/`.js` inline the shared
  `AUTO_CLOSE_REGEX_SOURCE` string via `JSON.stringify(...)` (the established
  `BARE_REF_REGEX_SOURCE` pattern — one source, embedded literal). A seam note in
  `matcher.ts` records that the gated A+B layer must consume this same evaluator.
- **Provenance honesty:** 1 confirmed instance (#2471→#2466) + 4 asserted-prior
  (undocumented). Never claim 5 confirmed.

## Shared module — `packages/core/src/autoclose/`

### `matcher.ts`

- `AUTO_CLOSE_KEYWORDS` — the nine keywords.
- `AUTO_CLOSE_REGEX_SOURCE` — canonical pattern string (flags `gi` applied by
  consumers). Shape:
  `\b(?:closed|closes|close|fixed|fixes|fix|resolved|resolves|resolve)\b\s*(?::\s*|\s+)(?:([owner/repo])#|#)(\d+)`
  - word-boundaried keyword (so `prefix`, `affixes`, `fixup` never match);
  - separator = a colon (optionally spaced) OR ≥1 whitespace (matches GitHub's
    `keyword #N`; declines `closed#88` with no separator, as GitHub does);
  - optional `owner/repo` qualifier before `#`; captured number.
- `findAutoCloseRefs(text): AutoCloseMatch[]` — `{ keyword, qualifier?, issue,
ref, index }` where `ref` is the normalized key (`owner/repo#N` lowercased, or
  `#N`).

Validated against the fixture set (see Test plan): the b8aa74a2 positive control
matches `#2466`; the subject-line parentheticals `(#2466) (#2471)` do NOT match
(no adjacent keyword).

### `receipt.ts`

- `AutoCloseReceipt` (schemaVersion, repo, prNumber, headSha, declaredCloseKeys:
  string[], corpusFindings: string[], generatedAt, note). `declaredCloseKeys` =
  the declared-intended-close set = normalized keys from
  `closingIssuesReferences` (GitHub API) ∪ structured-intent declarations.
- `parseDeclaredCloseIntent(text)` — the **structured intent channel** I design:
  a `<!-- totem-close: #N, owner/repo#M -->` HTML-comment marker (or a
  `Totem-Close: #N` trailer) an author adds to explicitly whitelist a genuinely
  intended close-keyword ref that is not otherwise linked. Primary channel stays
  `closingIssuesReferences` (the reframe: intended closures ride that or
  `gh issue close`).
- `scanPrCorpus({ title, body, commitMessages, closingIssuesReferences, repo })`
  → `{ findings, declaredCloseKeys, undeclared }`. D1 fails iff `undeclared`
  is non-empty. **Never scans comment bodies.**
- `reconcile(receipt | null, mergedBody, repo): ReconcileResult` — D2 logic.
  Statuses:
  - `clean` — body has NO auto-close matches (receipt irrelevant), OR every
    match's key ∈ `declaredCloseKeys`.
  - `anomaly` — matches present, receipt present, ≥1 match ∉ declared set.
    **The zero-allowed-set `[]` + closure-capable body is this case** (#2471
    specimen).
  - `missing-receipt` — matches present but receipt is null/absent (PRs merged
    before D1 existed; expired/undownloadable artifact). **Alert, never guess.**
  - `ambiguous-receipt` — receipt present but malformed / PR-mismatched, with
    matches present. **Alert, never guess.**
  - **No auto-reopen** anywhere — observation mode only (Tenet 9 sense→enforce
    gate: reopen arms later, behind positive+negative controls).

Quiet-by-default: a merged body with no close-keyword-adjacent ref is `clean`
regardless of receipt, so D2 does not perma-red every ordinary merge; it fails
loud precisely when a closure-capable body appears it cannot reconcile.

## Receipt persistence

- D1 writes `autoclose-receipt.json` and uploads it as a GitHub Actions artifact
  named `autoclose-receipt` with explicit `retention-days: 90`. D2 runs
  immediately at merge time (push-to-main), well within retention.
- D2 resolves the merged PR number from the squash subject `(#N)`, finds the D1
  `autoclose-guard` workflow run for the PR head SHA, and downloads the artifact.
  **Any** failure in that chain (no PR number, no run, no artifact, expired,
  unparseable) → `receipt = null` → `reconcile` returns `missing-receipt` if the
  body is closure-capable. That is the "PR merged before D1 existed → alert,
  never guess" branch, handled by one funnel.

## Workflow wiring

- **D1** `.github/workflows/autoclose-guard.yml` — `pull_request` → build core →
  `node tools/autoclose-pr.mjs` (imports the built `packages/core/dist/autoclose`),
  scans corpus, writes + uploads the receipt, fails on undeclared. Sibling to
  `lint.yml`.
- **D2** `.github/workflows/autoclose-postmerge.yml` — `push: branches:[main]` →
  build core → `node tools/autoclose-postmerge.mjs`, reconciles the merged HEAD
  body against the receipt, fails loud on anomaly/missing/ambiguous.
  **Observation mode: no reopen.** (Scripts live in `tools/` per the existing
  workflow-helper convention, e.g. `tools/vp-pr-cut-sensor.mjs`.)
