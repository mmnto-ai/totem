## Matcher policy (binding, all mechanisms)

- **Presence invariant, zero semantics.** Any GitHub close-keyword
  (`close/closes/closed`, `fix/fixes/fixed`, `resolve/resolves/resolved`)
  immediately adjacent to an issue reference is an anomaly — genuine, negated,
  quoted, or emphasized alike. **No negation parser.**
- **Qualified refs** (`owner/repo#N`) still auto-close → match them too.
- **Never scan issue/PR comment bodies** — comments never auto-close.
- **ONE shared evaluator** in `packages/core/src/autoclose/`; D1, D2, and C's
  template all consume it. C's rendered `.cjs`/`.js` inline the shared
  `AUTO_CLOSE_REGEX_SOURCE` string via `JSON.stringify(...)` — NOT a value-import
  (the cold-start rule #2339 forbids importing the heavy core barrel into a CLI
  command). It is a **local mirror drift-locked by a genuine parity test**
  (init.test.ts imports core's `AUTO_CLOSE_REGEX_SOURCE` and asserts both rendered
  templates inline it) — the same Tenet-4 licensed shape as the sealed
  `BARE_REF_REGEX_SOURCE`, not an independent copy. A seam note in `matcher.ts`
  records that the gated A+B layer must consume this same evaluator.
- **URL close form** (`https://github.com/owner/repo/(issues|pull)/N`, tolerant of
  a trailing `#issuecomment-…` fragment) is matched too, normalized to the same
  `owner/repo#N` key — undocumented for closing but empirically auto-closes (kimi
  BLOCKING-1; isaacs/github#1731, SAP/spartacus CONTRIBUTING).
- **Provenance honesty:** 1 confirmed instance (#2471→#2466) + 4 asserted-prior
  (undocumented). Never claim 5 confirmed.

## Shared module — `packages/core/src/autoclose/`

### `matcher.ts`

- `AUTO_CLOSE_KEYWORDS` — the nine keywords.
- `AUTO_CLOSE_REGEX_SOURCE` — canonical pattern string (flags `gi` applied by
  consumers).
  - word-boundaried keyword (so `prefix`, `affixes`, `fixup` never match);
  - separator = `(?:\s*:\s*|\s+)` — a colon (optionally spaced) OR ≥1 whitespace.
    The two alternatives are DISJOINT (no shared `\s*` prefix), killing the
    measured O(n²) backtracking of the earlier `\s*(?::\s*|\s+)` shape (kimi
    NON-BLOCKING-1): now linear on a whitespace run, ~0.2ms at the ~64KB body
    ceiling. Language-equivalent to the old shape (verified across the fixtures).
  - ref alternation (URL-first): URL `https://github.com/owner/repo/(issues|pull)/N`
    → groups 1+2; qualified `owner/repo#N` → groups 3+4; bare `#N` → group 5.
- `findAutoCloseRefs(text): AutoCloseMatch[]` — `{ keyword, qualifier?, issue,
ref, index }` where `ref` is the normalized key (`owner/repo#N` lowercased, or
  `#N`).

Validated against the fixture set (see Test plan): the b8aa74a2 positive control
matches `#2466`; the subject-line parentheticals `(#2466) (#2471)` do NOT match
(no adjacent keyword).

### `receipt.ts`

**Authorization is provenance-distinct (codex #3 — the circularity fix).** GitHub
DERIVES `closingIssuesReferences` from the PR body's own close keywords, so it
cannot authorize the same finding (a body keyword would self-whitelist). The SOLE
authorizing channel is the `totem-close` marker; `closingIssuesReferences` is
recorded as OBSERVED GitHub state (informational). **Author workflow: declare
every intended close with a `<!-- totem-close: #N -->` marker (or a
`Totem-Close: #N` trailer).**

- `AutoCloseReceipt` (schemaVersion **2**, repo, prNumber, headSha,
  **`declaredByMarker: string[]`** (authorizing), **`closingIssuesReferences:
string[]`** (informational), corpusFindings, generatedAt, note).
- `parseDeclaredCloseIntent(text)` — parses the `totem-close` marker/trailer. It
  is the ONLY authorizing channel. `stripIntentMarkers` runs before the finding
  scan so a marker (`totem-close: #N` contains `close: #N`) never self-flags
  (unit-tested).
- `scanPrCorpus(...)` → `{ ok, findings, declaredByMarker, closingIssuesReferences,
undeclared }`. D1 fails iff a finding is NOT marker-authorized. **Never scans
  comment bodies.**
- `reconcile(receipt | null, mergedBody, repo): ReconcileResult` — D2 logic;
  authorizes findings against `receipt.declaredByMarker` ONLY. Statuses:
  - `clean` — no auto-close matches (empty / trailer-only body), OR every match is
    marker-authorized.
  - `anomaly` — a closure-capable message with ≥1 UNAUTHORIZED ref. **The
    zero-allowed-set `[]` + closure-capable body is this case** (#2471 specimen).
  - `missing-receipt` — matches present but receipt null/absent. **Alert, never
    guess.**
  - `ambiguous-receipt` — receipt malformed (incl. a stale v1 shape) or
    PR-mismatched, with matches present. **Alert, never guess.**
  - `unexpected-body` — see the E-lever addendum below (body-presence signal).
  - **No auto-reopen** anywhere — observation mode only (Tenet 9 sense→enforce
    gate: reopen arms later, behind positive+negative controls).

Quiet-by-default: a merged body with no close-keyword-adjacent ref is `clean`
regardless of receipt, so D2 does not perma-red every ordinary merge; it fails
loud precisely when a closure-capable body appears it cannot reconcile.

## Receipt persistence

- D1 writes `autoclose-receipt.json` and uploads it as a GitHub Actions artifact
  named `autoclose-receipt` with explicit `retention-days: 90`. D2 runs
  immediately at merge time (push-to-main), well within retention.
- D2 resolves the merged PR number from the squash subject `(#N)`, finds the D1
  `autoclose-guard` workflow run for the PR head SHA, and downloads the artifact.
  **Any** failure in that chain (no PR number, no run, no artifact, expired,
  unparseable) → `receipt = null` → `reconcile` returns `missing-receipt` if the
  body is closure-capable. That is the "PR merged before D1 existed → alert,
  never guess" branch, handled by one funnel.

## E-lever addendum (strategy-claude dispatch 2026-07-21T0235Z)

The operator flipped all four governed repos to
`squash_merge_commit_title: PR_TITLE` + `squash_merge_commit_message: BLANK`
(the "E lever", executed 2026-07-21T02:30Z). BLANK stops GitHub composing a
squash body from `COMMIT_MESSAGES`; PR_TITLE makes the squash subject
deterministically the PR title (a surface D1 already scans). Two folded changes:

- **D1 asserts the merge-config posture and fails loud on drift.** A pure
  `evaluateMergeConfigPosture({title, message})` (core `autoclose/merge-config.ts`,
  unit-tested: conforming / title-drift / message-drift / both / absent) checks
  `title == PR_TITLE && message == BLANK`; thin `gh api repos/{repo}` glue in
  `tools/autoclose-pr.mjs` runs it FIRST and exits 1 on drift (Tenet 4
  parity-test rule applied to repo config — one settings-page click can silently
  revert it). Candidate standing row for the mmnto-ai/totem-strategy#482 parity
  manifest. D1's branch-commit-message scan is KEPT unchanged (config can drift;
  direct pushes to main still carry authored bodies).

- **D2 is body-presence-first.** Under BLANK the server composes no body, so a
  non-empty squash body is itself the posture signal (only a local
  `gh pr merge --body` override — the confirmed-vector class — a config-drift
  regression, or a non-squash merge produces one). `reconcile` now splits the
  commit message into subject + body, evaluates body-presence first, and runs the
  content scan over the whole message (a close-keyword in the PR_TITLE subject
  still auto-closes). Statuses: empty body + no undeclared ref → `clean` (quiet);
  undeclared close-keyword ref → `anomaly`/`missing-receipt`/`ambiguous-receipt`
  (hard alert, harm wins over the posture signal); **non-empty body with no
  undeclared close-keyword ref → new `unexpected-body`**.

  **Interpretation call** (the addendum's wording — "any non-empty body is the
  anomaly signal" — vs the "quiet-when-clean for a body carrying no
  close-keyword-adjacent ref" instruction, which the coordinator flagged as mine
  to resolve): a non-empty-but-keyword-free body causes NO accidental-closure
  harm, so it is surfaced as posture-drift EVIDENCE (`unexpected-body`) rather
  than a hard close-anomaly — D2 emits a NON-failing `::warning` (surfaced, not
  silent; not a false CI failure on a rare legitimate authored body), no reopen
  candidates. The hard exit-1 failures stay the keyword-bearing closure-harm
  cases. Rationale + the alternative (fail-loud on any non-empty body) recorded
  in the build report.

Everything else in the 0056Z contract stands (observation mode, no auto-reopen,
qualified-ref matching, no comment-body scanning). A+B remain un-gated.

## Squash-only posture (codex supplement 2026-07-21T0356Z)

`evaluateMergeConfigPosture` also asserts SQUASH-ONLY: `allow_squash_merge=true`,
`allow_merge_commit=false`, `allow_rebase_merge=false`. This is what LICENSES D2's
single-HEAD, `(#N)`-subject assumption — a rebase merge pushes several commits (an
earlier close-keyword commit would go unscanned when only the final SHA is read),
and a regular merge commit need not carry the squash subject (breaking receipt
correlation). Enabling squash-only is an OPERATOR settings action, routed
separately like the E lever; **until flipped, D1 reds loudly by design** (live
`gh api repos/mmnto-ai/totem` at review time reported all three methods enabled).

## Bootstrap: making D1 an actual required check (codex #1)

The `autoclose-guard` workflow creates the check but does NOT couple it to merge
admission — a red D1 run does not block merge until the operator enrolls the
check name in branch protection / a ruleset. **Check names only exist after a
first run, so this is an operator step AFTER the first PR run lands D1 on the
default branch.** Until enrolled, D1 is ADVISORY (D2's post-merge observation is
the backstop). This bootstrap/migration step is operator-routed (like E and
squash-only) and must be recorded in the PR body.

## Arming gate — falsifiable exit criteria (agy d, sensor-lens review)

D2 stays in observation mode (alerts only, no auto-reopen) until ALL of these are
met — the Tenet 9 sense→enforce gate made falsifiable rather than vibes-based:

1. **Positive control:** an end-to-end run on a sandbox repo proves D2 reopens a
   targeted issue on an undeclared merge.
2. **Negative control:** 30 days of zero false-positive alerts across all governed
   repos.
3. **Loop mitigation:** a hardwired reopen rate-limit (max 3 reopen attempts per
   issue per rolling 24h) so a webhook/state-drift cannot bot-loop.
4. **Emergency stand-down:** a standardized marker that forces D2 to stand down
   for a deliberately-undocumented merge-time close.

## D2 flow refinements (review round)

- **Content-scan-first** (agy b): D2 scans the merged message for refs BEFORE any
  receipt fetch — a clean commit makes ZERO `gh api` calls. Every non-clean status
  writes a `GITHUB_STEP_SUMMARY` block (visibility).
- **Trailer-strip** (strategy-claude 0330Z): RFC-822 trailer lines
  (`Co-authored-by:`/`Signed-off-by:` — `/^[A-Za-z-]+:\s/`) are stripped before the
  body-presence test, so a co-authored/dependabot squash (trailers survive BLANK)
  reads as `clean`, not `unexpected-body`. Fixture: the totem-strategy#948 body.
- **loadReceipt diagnostics** (agy a): the null contract is kept, but each broken
  link (PR lookup / run lookup / artifact download / JSON parse) logs a precise
  stderr warning naming the link. D2 also picks the LATEST D1 run for the head SHA
  (`sort_by(run_number)` desc) so an `edited`-PR rerun supersedes the earlier
  receipt.
- **D1 events** (codex #2): `autoclose-guard.yml` lists explicit
  `types: [opened, edited, synchronize, reopened]` — a title/body edit re-gates
  (under PR_TITLE a later title becomes the squash subject).
- **Pagination** (codex #4/#6): the branch-commit fetch (`--paginate`) and the
  GraphQL `closingIssuesReferences` (cursor loop) both run to exhaustion; a
  > 100-commit adapter test + a two-page GraphQL adapter test cover the seam.
