# Spec: #2141 seat-dir discovery + #2144 mail scan truncation (one lane — gate 4 of strategy#611)

Generated via `totem spec 2141 2144` (Gemini, run da927720b080) and then **corrected against ground truth** — the generated spec mis-located both bugs. Corrections are the record; the design below is authoritative.

## Problem statement

1. **#2141:** A new seat (`totem-codex`) that self-creates `.totem/orchestration/totem-codex/{outbox,journal}/` per the single-writer convention is invisible to `totem mail` — the self-agent set resolves from the frozen `COHORT_AGENT_MAP` basename map (`packages/core/src/orchestration-resolver.ts:157`), so mail addressed `to: totem-codex` is filtered out when the seat polls, and `totem mail send --to totem-codex` warns "not a known cohort agent." Roster ruling (strategy#611, operator 2026-06-11): seat admission at **repo+1 must touch ZERO surfaces** — the dir IS the registration.
2. **#2144:** `pollMail`'s unread-candidate set is _all_ outbox `.md` files cohort-wide minus the caller's `processed/` names. Mail addressed to OTHER agents never enters the caller's processed set, so the candidate set grows monotonically (>500 today; the truncation warning fires on every poll). The `MAX_SCAN = 500` cap (`packages/cli/src/commands/mail.ts:36`) slices newest-500 **by filename before the `to:` filter runs** — a self-addressed dispatch older than the newest 500 cohort-wide files is silently never opened. The per-poll warning is undirected (can't say whether anything addressed to YOU was dropped) and now fires constantly — alarm fatigue in the load-bearing coordination channel.

## Corrections vs the generated spec

- The generated spec located #2141 in `enumerateOutboxes` / a "hardcoded agent roster in `pollMail`". Ground truth: outbox enumeration is **already dir-derived**; the stale cache is `resolveSelfAgents`'s map layer + `knownCohortAgents()` in **core**, not cli.
- It said outbox files are `*.json`; they are `*.md` (ADR-098 dispatches).
- Its "skip processed/, journal/" fix misses the real cause: `processed/` and `journal/` are **not** in the scanned candidate set (the scan walks `outbox/` dirs only). The pollution is other-recipient mail in outboxes; growth in `processed/` affects only the exclusion-set build, not the cap.
- Its "newest-first before slice" is already implemented (mail.ts:433, GCA finding on #1971). The miss class survives it.

## Architectural context

- Tenet 20 (dirs ARE the state): the orchestration tree is the registration substrate. The map is a cache of it; #2141 is the cache going stale.
- ADR-106 § 3 / ADR-107: fail-open transport, scan-errors-always-warn, never-silent-drop (inv6; the #2118 forensics).
- `.totem/orchestration/` is **gitignored** (`.gitignore:65`) — local-only transport (Proposal 282). The dirs layer therefore reads only locally-created state; a malicious PR cannot inject seat dirs.
- `parseHeader` consumes at most `3 + MAX_HEADER_SEARCH_BYTES` of any file by construction (mail.ts:102,148) — a closing `---` beyond the window is already "absent." Bounded reads are therefore semantics-preserving.
- strategy#609 (canonicalize-event schema) rides this same substrate — see § Schema seam note.

## Implementation Design

### Scope

Mechanize seat discovery — the self-set and the known-recipient set derive from the `.totem/orchestration/` directory layout, with the basename map demoted to bootstrap fallback — and de-risk the scan horizon: bounded header reads, self-first scan ordering, a 10× cap raise, and a directed truncation warning. Explicitly NOT in scope: any persistent state or cache, frontmatter/schema changes, the #609 canonicalize event itself, `processed/` lifecycle changes, dashboard (totem-status) changes, deleting `COHORT_AGENT_MAP`.

### Data model deltas

| Delta                                                                                                        | Holds                            | Writes              | Reads                                | Invariants                                                                                                                                |
| ------------------------------------------------------------------------------------------------------------ | -------------------------------- | ------------------- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `SelfAgentResolution.source` gains `'dirs'` and `'dirs+map'` (core :221; duplicated union in cli mail.ts:63) | which precedence layer answered  | `resolveSelfAgents` | mail render line, `--json` consumers | additive literals; env/config/none cases byte-identical to today                                                                          |
| `knownCohortAgents(workspace?: string)` optional param (core :175)                                           | known recipient ids              | n/a (pure)          | `mailSend` content validation        | zero-arg ⟹ exactly the legacy map flatten (back-compat); with workspace ⟹ map ∪ dir-derived across `<workspace>/*/.totem/orchestration/*` |
| `MailSendOptions.workspace?: string` (cli)                                                                   | workspace override               | caller/CLI flag     | `mailSend`                           | additive; default `TOTEM_WORKSPACE` env, else `dirname(repoRoot)` — same resolution as `pollMail`                                         |

No new state containers. No reserved keys or sentinels. The resolver stays pure (no caching), matching its documented stance.

### State lifecycle

No new persistent, module-level, or cross-call state anywhere. All derivation is per-invocation `readdirSync`: one extra readdir of `<repoRoot>/.totem/orchestration/` in the dirs layer; one readdir per workspace repo inside `knownCohortAgents(workspace)` (~6 dirs today). Nothing crosses a lifecycle boundary.

### Resolution semantics (the map-vs-derive call)

New precedence: **env > config > (dirs ∪ map) > none.**

- dirs layer: `readdirSync(repoRoot/.totem/orchestration, { withFileTypes: true })`, keep directories only, exclude names starting with `.` or `_`, filter `isPathSafeAgentId`, sort. Any readdir failure ⟹ empty contribution (degrades to today's map behavior).
- **Union, not replacement:** dirs-first replacement would _regress_ partial-dir states — on a fresh clone (orchestration is gitignored) where only one seat has written, its roster siblings would vanish from the self-set. Union keeps every map-known sibling visible while admitting dir-only seats. `source` reports `'dirs'` (map contributed nothing novel), `'map'` (no usable dirs), or `'dirs+map'` (both contributed).
- env and config stay exclusive overrides (explicit beats derived; config's replace-not-augment contract is shipped and CHANGELOG-documented — Totem is not zero-user). **Codex F1 fold (warn-shape, the codex-sanctioned alternative):** when config answers and a safe seat dir exists that config omits, the resolution carries a loud warning naming the omitted seat(s) — `SelfAgentResolution.warnings?: string[]` (additive), surfaced through mail's warning stream. This is the in-repo mirror of strategy's `ecl-self-agent-binding` superset-of-dirs probe (strategy#618) and answers the operator's 2026-06-10 silent-unbind comment on #2141: config-exclusion is not a sanctioned decommission mechanism, and the suppression is never silent.
- `knownCohortAgents(workspace?)` traversal contract (codex F4): one level only — immediate repo children, immediate orchestration dirents, directories only, no symlink following, `isPathSafeAgentId` plus the same dot/`_`/node_modules exclusions as the mail scan. `--workspace` threads through `mail send` / `mail reply` for UX symmetry (recipient validation stays advisory, inv6).
- Prose copies of the map description (`init-templates.ts:753` signoff-skill content; resolver JSDoc) update in the same PR — the JSDoc on `COHORT_AGENT_MAP` names them keep-in-sync surfaces.

### Scan changes (#2144)

1. **Bounded header reads:** replace whole-file `readFileSync` (mail.ts:447) with a `3 + MAX_HEADER_SEARCH_BYTES`-byte read (`fs.openSync`/`readSync`). `parseHeader` already treats a closing delimiter beyond the window as absent, so semantics are preserved for every file that parses today; the byte-vs-UTF-16-unit nuance at the boundary can only convert an exotic >16KB-header parse into a LOUD mail-shaped warning, never a silent drop. **Codex F3 fold:** the byte boundary lives in the read helper — which returns a `bytesRead`/`truncated` signal feeding the parse-warning path — not in `parseHeader` string slicing; boundary fixtures (multi-byte char split at the read edge; closing `---` split across it) lock the loud-warning behavior.
2. **Self-first ordering (revised per codex F2):** bucket A = filename recipient token equals a self-agent or `broadcast`, matched POSITIONALLY immediately after the compact stamp (strategy note 2b — a slug word equal to a seat id must not mis-bucket; reserved tokens enumerated: `broadcast`, future `_`-class). A scans first, newest-first. Everything else — no-token and known-other-token alike — stays MERGED in global newest-first order, exactly today's baseline. The originally proposed B-before-C demotion is REJECTED: under a pre-parse cap, ordering becomes delivery — a known-other-token file carrying `to: self` in its header would die under unknown-token pressure where today's global order delivered it (codex F2). Net property: the token grants priority only; nothing ever ranks below today's baseline.
3. **Cap raise:** `MAX_SCAN` 500 → 5000. Worst case 5000 × ~16KB ≈ 80MB reads (realistic: ≈15MB) — inside session-start hook budget on local disk; the cap becomes a genuine OOM/latency backstop instead of the operating regime.
4. **Directed truncation warning:** when the cap trips, check the _dropped tail's filenames_ (zero I/O) for bucket-A membership; if any match, the warning names those files explicitly ("possible self-addressed mail beyond the scan horizon: …"). Generic warning unchanged otherwise. Strictly-better property: no new silence, new loudness only where it's actionable.

### Invariants to lock in via tests

1. A seat whose orchestration dir exists resolves as self with zero config/map presence — repo+1-zero-surfaces, mechanized (the totem-codex exhibit).
2. A map-listed sibling WITHOUT a dir stays in the self-set when another seat's dir exists (union; no partial-dir regression).
3. env and config each still shadow the dirs layer entirely.
4. Traversal-shaped or `_`/`.`-prefixed dir names never enter the self-set.
5. `knownCohortAgents()` zero-arg returns exactly the legacy flatten; with workspace, a dir-derived seat in ANY workspace repo is known (send warning gone for totem-codex).
6. A self-addressed dispatch older than >MAX_SCAN newer other-recipient files is still surfaced when its filename carries the self token (bucket A beats the cap); tokenless files retain today's global newest-first odds — never worse (codex F2).
7. `composeDispatch → bounded read → parseHeader` round-trips byte-identical to whole-file reads for every header that closes within the window.
8. A fixture whose header splits a multi-byte character at the read boundary produces the LOUD mail-shaped warning path — never a quiet parse-or-drop (strategy note 1; inv6/Tenet 4).
9. Truncation with a conventionally-named self-addressed file in the dropped tail fires the directed warning naming it; other-recipient-only tails fire the generic warning only.
10. A file whose filename token says other-agent but whose parsed `to:` says self is delivered at today's baseline or better — token grants priority only, never demotion (codex F2 regression: known-other token + `to: self` under heavy tokenless pressure ⟹ delivered).
11. Config-answers-while-omitting-a-present-safe-seat-dir carries a warning naming the omitted seat(s) (F1 warn-shape regression: config `[totem-claude]` + present `totem-codex/` dir ⟹ warning names totem-codex).
12. Resolver purity: a dir created between two calls is visible to the second (no caching).

### Schema seam note (strategy#609 — riding this design per the 2252Z dispatch)

The canonicalize-event settle adds an `event:`-class frontmatter field. Two seam facts from this lane: (a) the sensor/actuator pairing is one enumeration with two readers — any new field extends `composeDispatch` AND `parseHeader` together, and this design deliberately leaves `parseHeader`'s field set untouched (read-bounds only), so the seam stays clean for #609; (b) the filename recipient token is now load-bearing for scan ordering and the directed truncation warning — retire-dispatches emitted via `totem mail send` keep it automatically, so the schema settle should pin "actuator-emitted filenames carry the recipient token" as an explicit invariant, not an accident of the current slugger. **Accepted into the #609 settle** (strategy-claude on-issue, 2026-06-11T2335Z): MUST for actuator-emitted, SHOULD for hand-authored (missing/unknown token degrades to the merged pool — conformant-degraded, not violating); reserved tokens enumerated; positional matching.

1. **Q:** `source` label for the union layer — single `'dirs'` vs tri-state `'dirs' | 'map' | 'dirs+map'`?
   **Options:** single label (simpler union type, but lies when the map contributed); tri-state (honest discriminator; two extra literals across two files).
   **Recommendation:** tri-state — `source` exists to be honest about derivation.
2. **Q:** `MAX_SCAN` raised to 5000 vs removed in favor of a time-budget?
   **Options:** 5000 deterministic cap (testable, predictable) vs wall-clock budget (adaptive, but nondeterministic tests + a new failure mode).
   **Recommendation:** 5000 + the directed warning; revisit only if a real workspace approaches it.
3. **Q:** Does a bare agent dir (no subdirs yet) count as a seat in the dirs layer?
   **Options:** require `outbox/` or `processed/` to exist; any directory counts.
   **Recommendation:** any directory passing the filters counts — "the dir IS the registration," and requiring subdirs re-introduces a provisioning step at repo+1.
