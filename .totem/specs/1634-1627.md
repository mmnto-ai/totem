# Spec — #1634 + #1627 Bundle A: classifier extension + ledger hygiene

## Problem Statement

Two adjacent compile-worker defects share the same substrate (the `nonCompilable` ledger + the `reasonCode` classifier taxonomy established in #1598). Bundling them lets the second fix build directly on the first's invariants.

- **#1634 — semantic/multi-file classifier.** Compile-worker emits regex for lessons whose hazard lives at a scope the pattern cannot reach (multi-file contracts, closure-body AST, system-parameter-aware scoping, project-state-conditional semantics). ~50% of newly-compiled rules per postmerge cycle end up archived. Extends the #1598 `reasonCode` pattern with one consolidated `semantic-analysis-required` code.
- **#1627 — ledger hygiene.** Smoke-gate failures (`matches-good-example`, `missing-goodexample`, `pattern-zero-match`, `verify-retry-exhausted`, `pattern-syntax-invalid`, `missing-badexample`) currently land in `nonCompilable`. These are retry-pending (improve the prompt, re-roll the LLM, pattern may compile cleanly next cycle) — not permanently non-compilable. Writing them marks them as unfit for rules and blocks future retries.

## Architectural Context

Both extend ADR-091's Classify stage. #1634 adds a new classification dimension (semantic vs. surface); #1627 enforces the semantic distinction between "permanent" and "retry-pending" within the ledger. Sibling to #1598 (context-sensitive classifier, shipped 2026-04-23), which established the `reasonCode` enum pattern both tickets reuse.

## Files to Examine

1. `packages/core/src/compiler-schema.ts` — `NonCompilableReasonCodeSchema` enum (lines 198-220 post-#1598) and `CompilerOutputBaseSchema.reasonCode` narrow enum (currently `['context-required']`). Both need one new value.
2. `packages/core/src/compile-lesson.ts` — Pipeline 2 + 3 `!parsed.compilable` short-circuits (lines ~869 and ~1027 post-#1598). New classifier signal threads through the same branches.
3. `packages/cli/src/commands/compile.ts` — the `nonCompilableMap.set` call site at line 1260 (local skipped path) and line 1096 (cloud non-compilable path). The retry-pending guard lives here.
4. `packages/cli/src/commands/compile-templates.ts` — `COMPILER_SYSTEM_PROMPT` + `PIPELINE3_COMPILER_PROMPT`; #1598's "Context Constraints Classifier" section is the template pattern to extend.
5. `packages/core/src/compile-lesson.test.ts` + `packages/core/src/compiler-schema.test.ts` — mocked-LLM test surfaces.
6. `.totem/compiled-rules.json` — three existing `matches-good-example` entries at lines 12155, 12161, 12173 need one-off removal per #1627 Change 2.

## Technical Approach

### #1634 — Classifier extension

1. **Schema:** add `'semantic-analysis-required'` to `NonCompilableReasonCodeSchema` (before `'legacy-unknown'`) AND to the narrow LLM-emittable `CompilerOutputBaseSchema.reasonCode` enum (alongside `'context-required'`).
2. **Prompt:** extend both `COMPILER_SYSTEM_PROMPT` and `PIPELINE3_COMPILER_PROMPT` with a "Semantic Analysis Classifier" section describing the four sub-classes (multi-file, closure-body, system-param-aware, project-state-conditional) and when to emit. Position it as sibling to (not replacement of) the existing Context Constraints Classifier.
3. **Routing:** the existing Pipeline 2 + 3 `!parsed.compilable` branches already thread `parsed.reasonCode` through to the `skipped` result. Extending the match from `=== 'context-required'` to `∈ {context-required, semantic-analysis-required}` keeps the routing shape identical.

### #1627 — Ledger hygiene

1. **Code:** introduce a `LEDGER_RETRY_PENDING_CODES` set in `packages/core/src/compiler-schema.ts` (or sibling) enumerating the retry-eligible reason codes. Modify the `compile.ts:1260` (local) and `compile.ts:1096` (cloud) `nonCompilableMap.set` paths to guard on the set: retry-pending codes skip the ledger write; permanent codes land as today.
2. **Symmetric prune:** when a prior ledger entry exists for a lesson hash with a retry-pending code, delete it on the next compile run (mirror the existing `nonCompilableMap.delete` upgrade-target prune at line 1249). Prevents a retry-pending entry from persisting across compile cycles after a prompt improvement lets the lesson compile.
3. **Data cleanup:** archive the 3 existing `matches-good-example` entries from `.totem/compiled-rules.json`. Since these are ledger entries (not active rules), the cleanup is a direct JSON mutation + manifest refresh, not `totem lesson archive` (which operates on rules). Use `totem lesson compile --refresh-manifest` after the JSON edit to update the output hash.

## Acceptance

- [ ] `NonCompilableReasonCodeSchema` gains `'semantic-analysis-required'`.
- [ ] `CompilerOutputBaseSchema.reasonCode` narrow enum accepts `'semantic-analysis-required'` alongside `'context-required'`.
- [ ] Both compile prompts teach the LLM to detect the four semantic sub-classes and emit the new reasonCode.
- [ ] Local + cloud routing threads the new signal through `nonCompilableMap`.
- [ ] Regression: the five #1634 cited exhibits (including closure-body `par_iter_mut().for_each`, system-param `Local<Vec<T>>`, project-state-conditional `rayon`) would route to `nonCompilable` with `semantic-analysis-required` under the new heuristic.
- [ ] Smoke-gate failures (`matches-good-example`, `missing-goodexample`, `pattern-zero-match`, `verify-retry-exhausted`, `pattern-syntax-invalid`, `missing-badexample`) no longer append to `nonCompilableMap`.
- [ ] Regression: a lesson that fails the smoke gate compiles cleanly on re-run with an improved pattern (i.e., no permanent ledger pollution).
- [ ] One-off cleanup: 3 existing `matches-good-example` entries removed from `.totem/compiled-rules.json`; manifest refreshed.

## Implementation Design

### Scope

**Will do:** extend `NonCompilableReasonCodeSchema` and the narrow `CompilerOutputSchema.reasonCode` enum with one consolidated `'semantic-analysis-required'` value covering all four sub-classes from #1634. Extend both compile prompts with a Semantic Analysis Classifier section. Introduce a `LEDGER_RETRY_PENDING_CODES` set and guard the local + cloud `nonCompilableMap.set` paths on it. Sweep the 3 existing `matches-good-example` entries from `.totem/compiled-rules.json` with a manifest refresh. Tests: schema acceptance, Pipeline 2 + 3 routing of the new signal, ledger-guard rejection of retry-pending codes, ledger-prune on re-run after a prior retry-pending entry.

**Will NOT do:**

- **#1634 Change 2 (strengthened plausibility gate).** That proposes running compiled patterns against the consuming repo's non-example code to count over-matches beyond a threshold. This is literally ADR-091 Stage 4 Codebase Verifier work (1.16.0 headline, #1504). Scoping it out keeps this bundle tier-2 and preserves the Stage 4 design space for the dedicated work.
- **Four separate reason codes for #1634 sub-classes.** `context-multi-file`, `context-closure-body`, `context-system-param`, `context-project-state-conditional` would be more granular but: (a) harder for LLM to reliably discriminate, (b) enum sprawl, (c) #1598 precedent is one consolidated code with the sub-class in the `reason` prose. One `semantic-analysis-required` code it is.
- **Cloud-worker-specific behavior.** The cloud branch at `compile.ts:1096` gets the same routing guard so local and cloud cycles produce identical ledgers. No cloud-only carve-outs.
- **`totem lesson archive` for the 3 cleanup entries.** That command operates on active rules, not ledger entries. Direct JSON edit + `--refresh-manifest` is the intended path.
- **Changing existing reason codes or semantics.** All current codes retain their meaning; only the write-eligibility of retry-pending codes changes.

### Data model deltas

**Extended enums (two schemas, same new value):**

- `NonCompilableReasonCodeSchema` (persisted ledger enum) gains `'semantic-analysis-required'` before `'legacy-unknown'`.
- `CompilerOutputBaseSchema.reasonCode` (narrow LLM-emittable enum) extends from `z.enum(['context-required'])` to `z.enum(['context-required', 'semantic-analysis-required'])`.

**New module-level constant:**

- `LEDGER_RETRY_PENDING_CODES` — a `readonly Set<NonCompilableReasonCode>` or tuple, exported from `compiler-schema.ts` (sibling to `NonCompilableReasonCodeSchema` so the two taxonomies stay co-located).
  - **Holds:** the subset of reason codes that indicate retry-eligible failures (pattern-level or LLM-output-level transient defects).
  - **Members:** `'pattern-syntax-invalid'`, `'pattern-zero-match'`, `'verify-retry-exhausted'`, `'missing-badexample'`, `'missing-goodexample'`, `'matches-good-example'`.
  - **Non-members (permanent):** `'out-of-scope'`, `'context-required'`, `'semantic-analysis-required'`, `'security-rule-rejected'`, `'no-pattern-found'`, `'no-pattern-generated'`, `'legacy-unknown'`.
  - **Written by:** constant, never mutated.
  - **Read by:** `compile.ts:1260` and `compile.ts:1096` to guard the `nonCompilableMap.set` call.
  - **Invariants:** the set is a strict subset of `NonCompilableReasonCodeSchema.options`. A TypeScript test should verify membership against the enum so a future enum addition trips a compile-time reminder to classify.

### State lifecycle

No new persistent state. The existing `nonCompilableMap` (in-memory, per-compile-run) and `nonCompilable` array on `compiled-rules.json` (persistent) retain their shapes. The change is a write-eligibility filter: retry-pending codes skip the write, permanent codes write as today.

One subtle lifecycle concern: a lesson may appear in the ledger from a prior compile run under a retry-pending code (this is the current bug state, and 3 entries of this shape already exist). On the next compile run, the lesson re-enters the compile path, may produce a valid rule, and ends up under `newRules`. The stale retry-pending ledger entry must be pruned on success. Add a symmetric prune to the existing upgrade-target prune at `compile.ts:1249`:

```
// Stale-ledger prune (mmnto-ai/totem#1627). If the lesson previously
// landed in nonCompilable under a retry-pending code and now compiles
// cleanly, drop the stale entry so ledger reflects current truth.
if (nonCompilableMap.has(lesson.hash)) {
  const prior = nonCompilableMap.get(lesson.hash);
  if (prior && LEDGER_RETRY_PENDING_CODES.has(prior.reasonCode)) {
    nonCompilableMap.delete(lesson.hash);
  }
}
```

This runs only on `compiled` outcomes (not skipped — a retry-pending entry should survive if the retry also failed).

### Failure modes

| Failure                                                                 | Category               | Agent-facing surface                                                                    | Recovery                                                                                                  |
| ----------------------------------------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| LLM emits `semantic-analysis-required`                                  | runtime (expected)     | skipped with reasonCode `'semantic-analysis-required'`; ledger write (permanent class)  | none — lesson is semantically non-compilable                                                              |
| LLM emits `context-required` (existing #1598)                           | runtime (expected)     | skipped with reasonCode `'context-required'`; ledger write (permanent class, unchanged) | none                                                                                                      |
| Compile path produces retry-pending code (`matches-good-example`, etc.) | runtime (transient)    | skipped; **ledger write skipped under new guard**; lesson re-enters compile on next run | next compile run retries                                                                                  |
| Lesson with stale retry-pending ledger entry compiles cleanly           | runtime (recovery)     | rule lands in `newRules`; stale ledger entry pruned on the `compiled` branch            | n/a (healthy)                                                                                             |
| Lesson with stale retry-pending ledger entry fails compile again        | runtime                | skipped; ledger-write still guarded; stale entry remains until cleanup or success       | next compile run retries; manual curation via future `totem lesson clean-ledger` command if backlog grows |
| Schema validation fails on new enum value on old core reading new data  | init / version-skew    | Zod parse hard error at `CompiledRulesFileSchema` load                                  | minor-bump `@mmnto/core`; producers emit only post-bump                                                   |
| `LEDGER_RETRY_PENDING_CODES` drifts out of sync with enum additions     | init (authoring error) | compile-time type error via strict subset check                                         | fix at authoring time; TypeScript prevents the drift                                                      |

All rows either map to existing graceful paths or fail loud. No silent degradation introduced.

### Invariants to lock in via tests

1. `NonCompilableReasonCodeSchema.parse('semantic-analysis-required')` succeeds; `'legacy-unknown'` stays terminal.
2. `CompilerOutputSchema.parse({compilable: false, reasonCode: 'semantic-analysis-required', reason: '...'})` succeeds.
3. `CompilerOutputSchema.parse({compilable: false, reasonCode: 'verify-retry-exhausted'})` still fails — the narrow LLM enum rejects internal codes (locks in the #1598 invariant).
4. Pipeline 2 + Pipeline 3 with mocked `{compilable: false, reasonCode: 'semantic-analysis-required'}` response route to `skipped` with that reasonCode.
5. Anti-lazy: a compilable response with a pattern still compiles even when the lesson body contains multi-file / closure / system-param trigger keywords.
6. `LEDGER_RETRY_PENDING_CODES` is a strict subset of `NonCompilableReasonCodeSchema.options` (compile-time type assertion + a runtime test that iterates and asserts membership).
7. Ledger-guard: a `skipped` result with `reasonCode: 'matches-good-example'` does NOT land in `nonCompilableMap`. Mirror tests for each retry-pending code.
8. Ledger-guard positive: `skipped` with `reasonCode: 'out-of-scope'` (or `'context-required'`, or `'semantic-analysis-required'`) DOES land in `nonCompilableMap` (locks in that the guard is a filter, not a block).
9. Stale-ledger prune: a lesson with a prior `matches-good-example` ledger entry that now compiles cleanly results in the entry being removed from `nonCompilableMap` post-run.

### Open questions

- **Question 1:** one consolidated `semantic-analysis-required` code, or four narrow codes for the #1634 sub-classes?
  - **Options:**
    - (a) **One consolidated code.** Matches #1598 precedent. Sub-class lives in `reason` prose.
    - (b) **Four narrow codes:** `context-multi-file`, `context-closure-body`, `context-system-param`, `context-project-state-conditional`. More granular telemetry.
    - (c) **One umbrella + a `reasonSubclass` field.** Structured telemetry without enum sprawl; more schema surface.
  - **Recommendation:** **(a) One consolidated code.** Simplest LLM contract, consistent with #1598. Telemetry on sub-class can happen in a later ticket once there's postmerge evidence that the sub-class distinction matters for triage.

- **Question 2:** should the ledger-hygiene guard be upstream (in `compile-lesson.ts` — return a non-ledger-write signal) or downstream (in `compile.ts` — filter at the `nonCompilableMap.set` call site)?
  - **Options:**
    - (a) **Downstream filter in `compile.ts`.** `compile-lesson.ts` keeps returning the full `skipped` shape; the CLI call site consults `LEDGER_RETRY_PENDING_CODES` before writing. Minimal surface change.
    - (b) **Upstream signal in `compile-lesson.ts`.** Add a `skipped-retry-pending` status variant on `CompileLessonResult`. More type-safe, but changes the core contract and cascades through every test that switches on the result.
  - **Recommendation:** **(a) Downstream filter.** The retry-pending-vs-permanent distinction is a ledger-semantic concern, not a core-compile concern. `compile-lesson.ts` correctly returns what happened; `compile.ts` decides what to persist. Matches the existing separation where core emits results and CLI applies persistence policy.

- **Question 3:** data cleanup — commit-time edit to `compiled-rules.json` plus `--refresh-manifest`, or wait for the stale-prune logic in Change 2 to clear it naturally on the next compile?
  - **Options:**
    - (a) **Commit-time cleanup.** Direct JSON edit removes the 3 entries; `pnpm exec totem lesson compile --refresh-manifest` updates the manifest hash. Ledger is clean at merge.
    - (b) **Defer to natural prune.** Let Change 2's stale-prune logic remove the entries on the next compile run that touches those lessons. Requires the lessons to be edited or the manifest to be forced; they may not naturally re-enter compile scope soon.
  - **Recommendation:** **(a) Commit-time cleanup.** The 3 entries are the literal defect #1627 was filed for. Leaving them to a future natural prune means the PR closes with the symptom still visible in the ledger. Direct edit is cheap and verifiable.

- **Question 4:** should the tests for ledger-guard live in `compile-lesson.test.ts` (core) or a new `compile.test.ts` (cli)? The guard logic is in `compile.ts`, but `compile.ts` already has extensive test infrastructure.
  - **Options:**
    - (a) **CLI-level tests.** Test the full compile-run loop with mocked `compileLesson` returning each retry-pending code; assert `nonCompilableMap` state post-run.
    - (b) **Unit-level tests on the guard function.** Extract the guard logic as a pure helper (e.g., `shouldWriteToLedger(result: CompileLessonResult): boolean`), unit test it in isolation; CLI-level integration test stays thin.
  - **Recommendation:** **(b) Unit-level tests on a pure helper.** Cheaper, more readable, covers every enum value mechanically. Add one integration test at the CLI level to assert the wiring.
