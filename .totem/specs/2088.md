# Spec — #2088 · `totem init --doctrine` (Proposal 292 S1, parity-manifest distribution)

## Problem statement

`totem doctor --parity` honest-absent-SKIPs in every consumer because the strategy-owned `parity-manifest.yaml` never reaches them and `orient.parityManifest` is unset. Proposal 292 §10 settled the distribution as **emitter-B**: strategy publishes the manifest snapshot + `strategy-doctrine.lock` as a private package (`@mmnto-ai/strategy-doctrine`, GitHub Packages); totem-core's entire job is to **wire the config pointer** when that pin is present. This is the decouple gate's missing half (strategy#545) and liquid-city's unblock.

## Architectural context (verified against code)

- **No schema change.** `OrientConfigSchema.parityManifest: z.string().optional()` already ships (`packages/core/src/config-schema.ts:259`), resolved relative to the config/repo root. The reader, the SKIP, the `not-found`/`unparseable` warns, and the `configured` self-gate all shipped in 1.53.7 (`doctor-parity.ts`).
- **Config-write prior art** = `packages/cli/src/commands/link.ts:101-140`: string-splice into `totem.config.ts` (find anchor → bracket-match → insert), fail-loud `TotemParseError` when the anchor is missing. No AST. `install.ts:406` also mutates config. S1 generalizes that pattern to the `orient` field.
- **The hard part** is that `orient` has three consumer shapes (none / present-without-`parityManifest` / already-set), where `link` only inserts into a single `targets: [` array.

## Implementation Design

### Scope

`totem init --doctrine` (non-interactive sub-path of `init`): detect the `@mmnto-ai/strategy-doctrine` pin in the consumer's `node_modules`; if present, write **only** `orient.parityManifest` → the pin's `parity-manifest.yaml` path into the existing `totem.config.ts`, idempotently and preserving all other config. It will **NOT** add the dependency to `package.json`, fetch anything over the network, mutate the manifest, touch the reader, or change the schema. No pin present ⇒ honest-absent: print guidance, write nothing, exit 0.

### Data model deltas

None. `orient.parityManifest` (string, optional) already exists and is the only field written. No new type, field, or state container. The written **value** is a repo-root-relative path string: `node_modules/@mmnto-ai/strategy-doctrine/parity-manifest.yaml`.

### State lifecycle

The single mutation is the on-disk `totem.config.ts` (persistent, repo-committed by the user). Created/owned by the user via `totem init`; this flag adds exactly one field on an explicit invocation. No in-memory/session state; no flag crosses a lifecycle boundary. Idempotent: a second run with the field already set is a no-op (detect-and-skip, mirroring `link.ts`'s "already linked" guard).

### Failure modes

| Failure                                                                     | Category             | Agent-facing surface                                                           | Recovery                                                |
| --------------------------------------------------------------------------- | -------------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------- |
| Pin (`@mmnto-ai/strategy-doctrine`) absent from node_modules                | runtime / expected   | warning + guidance ("add the pin dep, then re-run"); writes nothing; exit 0    | user adds the dep, re-runs — honest-absent preserved    |
| No `totem.config.ts` in cwd                                                 | runtime              | hard error (`TotemConfigError`, mirrors `link.ts:39`)                          | run `totem init` first                                  |
| Config present but `orient.parityManifest` already set                      | runtime / idempotent | info ("already wired"); no write                                               | none needed                                             |
| Config too exotic to splice safely (no parseable anchor / nested ambiguity) | permanent            | hard error (`TotemParseError`) **+ print the exact snippet to paste manually** | user pastes the one-liner; never a silent/partial write |
| Pin resolves but `parity-manifest.yaml` missing inside it                   | runtime              | warning (delegates to the reader's existing `not-found` warn at doctor-time)   | strategy republish; not totem-core's recovery           |

No row is "silent degradation" — every branch either writes the correct field, no-ops idempotently, or fails loud with a manual-paste fallback (Tenet 4).

### Invariants to lock in via tests

- Pin absent ⇒ config is **byte-unchanged** and exit is 0 (honest-absent, not failure).
- Pin present + no `orient` key ⇒ a new `orient: { parityManifest: '<path>' }` field is added and the file still parses under `TotemConfigSchema`.
- Pin present + existing `orient: { projectNumber: N }` ⇒ `parityManifest` is added **without clobbering** `projectNumber`.
- Already-set ⇒ idempotent no-op (no duplicate field, no rewrite).
- Unspliceable config ⇒ throws `TotemParseError` **and** the manual snippet appears in output; the file is never left partially written.
- The written path resolves (via the existing reader) to a real manifest under the installed pin → reader reports `pass`, not `not-found`.

### Open questions

- **Q1 — auto-write vs detect-and-print.** Auto-splice into `totem.config.ts` (matches `link.ts`; fragile on exotic configs, mitigated by fail-loud + manual-snippet fallback) **vs** always just print the snippet for the user to paste (zero mutation risk, but not the "writes the config pointer" shape we agreed). **Recommendation:** auto-write with the fail-loud manual-snippet fallback — honors the agreed opt-in shape and the #2008 "preserve user config" discipline, and the fallback makes the fragile case safe.
- **Q2 — literal path vs `require.resolve`.** Store the literal `node_modules/@mmnto-ai/strategy-doctrine/parity-manifest.yaml` (matches strategy's locked target; works for pnpm's direct-dep top-level symlink; zero reader change) **vs** resolve via `require.resolve(...)` and store the resolved relative path (robust to exotic layouts). **Recommendation:** literal path — it's what strategy locked and needs no reader change; `require.resolve` is the hedge only if a layout bites.
- **Q3 — command surface.** `totem init --doctrine` (agreed) **vs** a `link`-family command (`totem link --doctrine`), given `link.ts` is the literal prior art for "wire a path into config." **Recommendation:** keep `totem init --doctrine` per the cohort agreement; note the `link` alternative only for the record.

## Cost estimate

**Small** (consistent with strategy's "collapsed surface" read). One new flag branch in `init` (non-interactive), one config-splice helper generalizing `link.ts`'s pattern to the `orient` object (the only non-trivial code), pin detection (`fs.existsSync`/`require.resolve`), and a focused test matrix (the six invariants above — three config shapes × present/absent + the fail-loud case). No schema, no reader, no new types. Estimate: ~1 PR, the config-splice helper + its edge-case tests being the bulk. The weight of 292's distribution half is strategy's parallel lane (publish CI, lock shape, provenance bind, currency contract), none of which gates this.
